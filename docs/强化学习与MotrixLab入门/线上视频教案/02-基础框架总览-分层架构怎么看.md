# 02 基础框架总览：分层架构怎么看

## 课程信息
- **课程名称**：MotrixLab 入门精品课程
- **本章时长**：30 分钟
- **本章主题**：基础框架总览，分层架构怎么看

## 本章大纲
- MotrixLab 的整体分层设计
- 用户接口层、训练算法层、环境实现层、物理仿真层
- 各层之间的关系与职责划分

## 讲授目标
讲完这一章，学生应该能够：
1. 用分层视角理解 MotrixLab 的整体结构。
2. 说清楚用户接口层、训练算法层、环境实现层、物理仿真层分别负责什么。
3. 知道一次训练任务在各层之间大致如何流转。

---

## 0-4 分钟：开场与承接

上一章我们讲了 MotrixLab 的定位：它是一个面向机器人强化学习的开发平台，帮助我们把环境、仿真、训练、奖励和 Demo 分析组织起来。

这一章我们进一步看它的基础框架。

学习一个工程框架，最重要的不是先记住所有类名，而是先知道系统分成哪些层。因为分层决定了我们以后怎么读代码、怎么定位问题、怎么扩展功能。

如果没有分层视角，看到一个训练任务失败，我们可能会到处改：改训练脚本、改奖励、改模型文件、改算法参数。但如果有分层视角，我们会先判断问题属于哪一层。

比如：
- 命令行参数错了，可能是用户接口层的问题。
- PPO 配置不合理，可能是训练算法层的问题。
- 奖励计算异常，可能是环境实现层的问题。
- 机器人仿真不稳定，可能是物理仿真层的问题。

所以这一章我们要建立一个非常实用的框架阅读方法：**先看层，再看职责**。

---

## 4-10 分钟：MotrixLab 的整体分层设计

MotrixLab 的整体结构可以先概括成四层：

```
用户接口层
训练算法层
环境实现层
物理仿真层
```

这四层不是孤立的，而是从上到下协作。

- **用户接口层**：负责接收用户命令，比如训练、测试、查看模型。
- **训练算法层**：负责策略学习，比如采样、计算损失、更新网络参数。
- **环境实现层**：负责定义任务本身，比如观测、动作、奖励、终止条件。
- **物理仿真层**：负责底层物理计算，比如机器人关节、碰撞、接触和动力学。

分层带来的好处是职责清晰：
- 环境开发者主要关心任务，不需要深入修改 PPO 训练器。
- 算法开发者主要关心训练过程，不需要知道每个机器人任务的细节。
- 使用者通过命令行启动任务，不需要手动拼接所有底层模块。

## 10-18 分钟：四层职责详解

### 第一层：用户接口层

用户接口层是学生最容易接触到的一层。常见入口包括 `train.py`、`play.py`、`view.py` 这类脚本。

它们的职责不是实现所有任务逻辑，而是**负责启动流程**。比如：
- 读取命令行参数
- 判断要训练哪个任务
- 选择训练框架
- 加载配置
- 创建环境
- 启动训练或测试流程

所以以后如果你想知道"这个任务怎么跑起来"，应该先看用户接口层。

### 第二层：训练算法层

这一层关心的是强化学习算法如何工作。比如使用 PPO 时，它要完成数据采样、优势估计、策略损失、价值损失、熵奖励、参数更新等工作。

在 MotrixLab 中，训练算法层可能对接不同训练后端，例如 SKRL 或 RSLRL。不同后端内部实现不同，但它们都需要和环境交互。

这里要强调一个接口思想：**训练算法不应该直接关心机器人细节**。它只需要环境返回标准强化学习信息：观测、奖励、终止标志等。

### 第三层：环境实现层

这是本课程后面最重要的一层。环境实现层主要包括 `EnvCfg`、`Env` 和 `Reward`。

- `EnvCfg`：定义环境参数。比如模型文件、仿真步长、episode 长度、动作范围、奖励权重等。
- `Env`：实现任务逻辑。比如如何接收动作，如何推进仿真，如何构造观测，如何判断任务结束。
- `Reward`：负责生成学习信号。它告诉智能体当前行为是否更接近任务目标。

很多训练失败的问题，其实都发生在这一层。比如观测缺关键信息、动作尺度不合理、奖励方向错误、终止条件过严。

### 第四层：物理仿真层

这一层负责让机器人真的"动起来"。它处理机器人模型、关节、碰撞、接触、质量、惯性、动力学和仿真步进。

如果机器人一开始就抖动、穿模、爆炸，或者动作完全不符合物理直觉，不能只看奖励函数，也要检查物理仿真层。

## 18-25 分钟：各层之间的关系

接下来我们把四层串成一个完整训练流程。

一次训练通常从用户命令开始。用户在命令行指定要训练哪个任务、使用哪个训练框架、训练多少步、是否开启可视化等。

用户接口层收到命令以后，会解析参数，然后通过注册和配置系统找到对应任务。

接着训练算法层被初始化。它会创建策略网络、价值网络、优化器和训练器。

然后环境实现层被创建。环境根据 `EnvCfg` 加载模型和任务参数，准备好观测、动作、奖励和终止逻辑。

训练开始后，算法输出动作，环境接收动作并交给物理仿真层推进。仿真层返回新的物理状态，环境再根据状态计算观测和奖励，训练算法拿到反馈后更新策略。

可以把这一过程总结成：

```
用户命令
-> 训练入口
-> 配置和注册
-> 环境创建
-> 算法采样
-> 物理仿真
-> 奖励反馈
-> 策略更新
```

这个流程有助于排查问题：
- 如果环境创建失败，要看配置、注册和模型路径。
- 如果动作没有效果，要看动作映射、控制接口和物理仿真。
- 如果 reward 异常，要看奖励函数和环境状态。
- 如果训练曲线不收敛，要综合看奖励、观测、动作和算法参数。

---

## 25-28 分钟：课堂互动

三个问题帮助学生把"分层"变成实际排查能力：

1. 如果我想知道一个任务从哪里启动，应该先看哪一层？→ **用户接口层**
2. 如果我想修改任务目标和奖励，主要看哪一层？→ **环境实现层**
3. 如果机器人在仿真里刚开始就不稳定，可能只改 PPO 学习率能解决吗？→ 通常不能，应该先看物理仿真层和环境参数。

---

## 28-30 分钟：本章总结

这一章我们讲了 MotrixLab 的整体分层。

- **用户接口层**：负责启动和命令解析
- **训练算法层**：负责策略学习
- **环境实现层**：负责定义任务、观测、动作、奖励和终止条件
- **物理仿真层**：负责底层物理计算

各层之间通过训练流程连接起来：用户启动任务 → 算法和环境交互 → 环境调用仿真 → 仿真返回状态 → 环境计算奖励 → 算法更新策略。

**本章最重要的一句话：** 读 MotrixLab 项目时，先判断文件属于哪一层，再看它承担什么职责。
