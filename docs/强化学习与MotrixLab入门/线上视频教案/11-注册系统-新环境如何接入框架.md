# 注册系统：新环境如何接入框架

## 注册系统的作用

MotrixLab 有两套注册表，分工不同：

1. **`motrix_envs.registry`（仿真环境）**
   - 用字符串 `name`（例如 `"cartpole"`）绑定：环境配置类（`EnvCfg` 子类）和环境实现类（`ABEnv` 子类）
   - 对外统一入口是 `make(name, sim_backend=..., env_cfg_override=..., num_envs=...)`
   - 作用：把「环境名字」解析成可运行的环境对象

2. **`motrix_rl.registry`（RL 训练配置）**
   - 用同一个环境名字 `env_name`，在框架名（当前固定为 `"skrl"`）和训练后端（`"jax"` / `"torch"`）下注册 PPO 等训练超参的 dataclass
   - `default_rl_cfg(env_name, "skrl", backend=...)` 会按后端优先选专用配置
   - 作用：把「环境名字 + 训练后端」解析成默认 RL 超参

## 环境配置、环境实现、RL 配置的注册方式

以 `cartpole` 为最小闭环示例（名字在三处一致）：

**推荐目录结构（与仓库一致）：**
```
motrix_envs/src/motrix_envs/basic/cartpole/
├── cfg.py              # ① 环境配置 + @registry.envcfg("cartpole")
├── cartpole_np.py      # ② 环境实现 + @registry.env("cartpole", "np")
├── cartpole.xml        # MotrixSim 模型
└── __init__.py         # from . import cartpole_np  # 触发注册

motrix_rl/src/motrix_rl/cfgs.py
└── class basic:
        @rlcfg("cartpole")   # ③ RL 配置，名字必须与上面一致
        class CartPolePPO(PPOCfg): ...
```

三处字符串必须相同：`"cartpole"`。

`train.py` 里 `--env cartpole` 会依次走：
1. `motrix_rl.registry.default_rl_cfg("cartpole", "skrl", backend=...)`
2. `motrix_envs.registry.make("cartpole", sim_backend=..., num_envs=...)`

---

### 环境配置（`EnvCfg` + `@registry.envcfg("名字")`）

**写什么：**
`cfg.py` 里定义继承 `EnvCfg` 的 dataclass，并挂上装饰器：
```python
@registry.envcfg("cartpole")
@dataclass
class CartPoleEnvCfg(EnvCfg):
    model_file: str = model_file
    reset_noise_scale: float = 0.01
    max_episode_seconds: float = 10
    render_spacing: float = 2.0
```

**基类 `EnvCfg` 已提供、一般不用重复写的字段：**
仿真与 episode 的通用字段（模型文件路径、sim_dt、ctrl_dt、max_episode_seconds、max_episode_steps、render_spacing 等）。

**`validate()` 何时执行：**
注册时不会调用 `validate()`；只有在 `registry.make()` 里实例化配置后才会调用：
```python
env_cfg = meta.env_cfg_cls()
if env_cfg_override is not None:
    for key, value in env_cfg_override.items():
        ...
env_cfg.validate()
```

**装饰器做了什么：**
`@registry.envcfg("cartpole")` 在类定义被执行时调用 `register_env_config`，在全局表 `_envs` 里写入：
```python
_envs["cartpole"] = EnvMeta(env_cfg_cls=CartPoleEnvCfg, env_cls_dict={})
# 此时还没有实现类
```

### 环境实现（`ABEnv` / `NpEnv` + `@registry.env("名字", "np")`）

**写什么：**
```python
@registry.env("cartpole", "np")
class CartPoleEnv(NpEnv):
    # ...
```

当前框架里 `sim_backend` 只支持 `"np"`。

**装饰器顺序与依赖：**
必须先有 `@envcfg`，再有 `@env`：
```python
if name not in _envs:
    raise ValueError(f"Environment '{name}' is not registered. Please register the config first.")
```

`@env` 会把实现类放进：`_envs["cartpole"].env_cls_dict["np"] = CartPoleEnv`

**`NpEnv` 基类已经帮你做的事：**
在 `__init__` 里：
- `mtx.load_model(cfg.model_file)` 加载 XML
- 设置 `model.options.timestep = cfg.sim_dt`
- 保存 `num_envs`、`render_spacing`

你需要在子类 `__init__` 里（在 `super().__init__` 之后）定义：
- `observation_space` / `action_space`
- 与模型相关的缓存（如 `_init_dof_pos`）

**必须实现的三个抽象方法：**
1. `apply_action(actions, state)` — 将动作写进仿真
2. `update_state(state)` — 读取新状态（obs / reward / terminated）
3. `reset(data)` — 重置环境并返回初始 obs

**一步 `step` 的内部顺序：**
```python
def step(self, actions):
    self._prev_physics_step()                    # 清空上一步的 reward/terminated
    self._state = self.apply_action(actions, self._state)  # 将动作写进仿真
    self.physics_step()                          # 物理步
    self._state = self.update_state(self._state) # 读新状态 obs/reward/terminated
    self._state.info["steps"] += 1
    self._update_truncate()                      # 按 max_episode_steps 设 truncated
    self._reset_done_envs()                      # 已结束的 env 做 reset()
```

**如何让装饰器执行：import 链**

装饰器只在模块被 import 时运行。cartpole 的链：
```
import motrix_envs
  → basic/__init__.py: from . import cartpole, ...
    → cartpole/__init__.py: from . import cartpole_np
      → 执行 @envcfg（cfg 在 cartpole_np 里 from .cfg import，cfg 模块先加载）
      → 执行 @env
```

新任务：在对应分类的 `__init__.py` 里加 `from . import your_env_np`，否则注册了也找不到。

### RL 配置（`motrix_rl.cfgs` 里 `@rlcfg("名字")`）

**写什么：**
在 `motrix_rl/cfgs.py` 的命名空间类里：
```python
class basic:
    @rlcfg("cartpole")
    @dataclass
    class CartPolePPO(PPOCfg):
        max_env_steps: int = 10_000_000
        check_point_interval: int = 500
        policy_hidden_layer_sizes: tuple[int, ...] = (32, 32)
        value_hidden_layer_sizes: tuple[int, ...] = (32, 32)
        rollouts: int = 32
        learning_epochs: int = 5
        mini_batches: int = 4
```

**`@rlcfg` 的行为：**
```python
def rlcfg(env_name: str, backend: str = None):
    def decorator(cls):
        backends = ["jax", "torch"] if backend is None else [backend]
        for b in backends:
            _register_rlcfg(env_name, "skrl", b, cls)
        return cls
```

未指定 backend → 同时注册到 skrl/jax 和 skrl/torch。

**训练时如何用这份配置：**
```python
rlcfg = registry.default_rl_cfg(env_name, "skrl", backend="jax")
if cfg_override is not None:
    rlcfg = rlcfg.replace(**cfg_override)
# ...
env = env_registry.make(self._env_name, sim_backend=self._sim_backend, num_envs=rlcfg.num_envs)
```

## 如何让新任务被框架"自动识别"

### "自动识别"是什么

MotrixLab 不会扫描目录、读配置文件列表。所谓"自动识别"，指的是：Python 在 import 某个包/模块时，执行类定义上的装饰器，把条目写进内存里的两个字典；之后脚本根据字符串名字去查字典。

因此新任务要能被 `train.py` / `view.py` / `play.py` 用上，必须同时满足：
1. 装饰器已执行（模块被 import 过）
2. CLI 里的 `--env` 字符串与注册名完全一致
3. 训练/回放时：环境注册 + RL 注册都存在（`view` 只需环境注册）

### 接入新任务的具体步骤

假设新任务注册名为 `my-task`：

**步骤 1：实现并装饰**
- `@registry.envcfg("my-task")`
- `@registry.env("my-task", "np")`

**步骤 2：接上 import 链（核心）**

任选一种：

A. 挂到现有分类（推荐）
```
# motrix_envs/basic/__init__.py 增加：
from . import cartpole, ..., my_task  # noqa: F401

# motrix_envs/basic/my_task/__init__.py
from . import my_task_np  # noqa: F401
```

B. 新建顶层分类
```
# motrix_envs/__init__.py
from . import basic, locomotion, manipulation, my_category  # noqa: F401
```

**步骤 3：RL 配置 + 保证会被加载**
在 `motrix_rl/cfgs.py`（或你新建但被 `motrix_rl/__init__.py` import 的模块）里：
```python
@rlcfg("my-task")
@dataclass
class MyTaskPPO(PPOCfg):
    ...
```

**步骤 4：用同一字符串跑命令**
```bash
uv run scripts/train.py --env my-task
uv run scripts/view.py --env my-task
```

注意：名字区分大小写；`my-task` 与 `my_task` 是两个不同 key。

### 各入口脚本如何"认出"任务名

- **`train.py`**：先加载 rl cfg 注册 RL，再 `default_rl_cfg(...)` 获取配置，最后 `env_registry.make(...)` 建环境
- **`view.py`**：只走 `registry.make(env_name, ...)`，不查 RL 表
- **`play.py`**：同 `train`，通过 `ppo.Trainer`；策略文件按环境名找 checkpoint：`runs/{env_name}/` 下找最新 `best_agent.*`
