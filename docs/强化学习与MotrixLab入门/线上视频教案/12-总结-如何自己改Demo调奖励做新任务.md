# 总结：如何自己改 Demo、调奖励、做新任务

---

## 一、修改现有 Demo 的思路（调整奖励和配置参数）

按照之前第 11 部分讲的思路：

### 1. 修改运动速度
- 修改 walker/humanoid 的 `move_speed` 参数即可
- 也可以修改 RL cfg 的超参数去体会训练效果的改变

### 2. 调整 GO1 的步态
可以把平稳的 gait 变成蹦蹦跳跳前进的。

**核心思路**：把"平稳约束"放松一点，把"腾空节奏"强化一点，但保留速度跟踪不丢。
- 减少对 GO1 加速度变化和姿态变化的限制，允许姿态上高度更大的变化
- 前进速度的奖励不能丢，甚至要稍微增加这部分权重
- 考虑到关节的限制，不然会限制腿部的摆动幅度

## 二、如何基于框架开发新任务

以配置新的环境 `go1-flat-terrain-hop` 为例（不新增 XML/机器人模型）：

### 修改 1 — 新增环境 cfg
添加 `@registry.envcfg("go1-flat-terrain-hop")` 的 `Go1HopNpEnvCfg`（继承自 `Go1WalkNpEnvCfg`）

### 修改 2 — 注册环境实现（复用现有类）
在现有环境类上也注册新名字，在类装饰器附近增加：
```python
@registry.env("go1-flat-terrain-hop", sim_backend="np")
```
这样复用 `Go1WalkTask` 的实现，不改行为逻辑，只改变 cfg。

### 修改 3 — 注册/配置训练
为 `go1-flat-terrain-hop` 增加 `@rlcfg("go1-flat-terrain-hop")` 的训练配置类（SKRL 与 RSLRL），可以复用 flat 的配置并微调。

## 三、后续学习与实践建议

### 做好记录和版本管理

每次跑实验之前，在项目目录里建一个文件夹，随手写一个 `exp_log.md`：

```
- 今天改了什么参数（比如把速度奖励权重从 1.0 调到 2.0）
- 预期会看到什么变化（希望机器人步幅变大）
- 实际跑下来看到什么效果（1000步后开始同手同脚）
```

这个简单的记录，积累几个星期，对任务的直觉就会飞速增长——开始把参数改动和行为变化之间建立了因果映射，不再是黑盒调参。

### 多写注释
帮助理解整体的项目框架。

### 可视化调试
训练期间可以并行只开 1-2 个环境渲染出来，单独观察一个机器人的行为，看到训练过程中每一次尝试后的机器人是否朝着奖励策略的方向改进。

### 加载自己的机器人模型

MotrixLab 内部有一个很大的优势——MotrixSim 引擎的优化极大降低了仿真的硬件门槛，算力消耗仅为国外同类产品的十分之一。即使使用普通的电脑，也能跑得动训练任务。

**具体步骤**：
1. 找一个现成的机器人 URDF
2. 复制一份距离最近的目标最近的现有任务
3. 替换模型路径、关节名、默认姿态、PD 参数等信息，在注册表里用 `@registry.envcfg("my-robot")` 注册新名字
4. 先别调奖励，先跑通环境——用 `view.py` 看一下模型能不能正常导入、关节能不能动、碰撞体对不对
5. 等环境能稳定交互了，再开始奖励调参，逐步让机器人学会站立、行走

**这个路径一旦走通，你就不再是强化学习的"旁观者"，而是能独立设计和闭环的开发者了。** MotrixLab 也会成为趁手的工具，后面就可以尝试 sim2real，将自己在仿真中设计的应用在真机中啦！
