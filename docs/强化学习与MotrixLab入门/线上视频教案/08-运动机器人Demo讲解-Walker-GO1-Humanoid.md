# 运动机器人 Demo 讲解：Walker / GO1 / Humanoid

## 核心目标

在 MotrixLab 的 `motrix_envs` 中，提供了多种预置的运动机器人环境，最典型的是：
- **Walker**（双足步行者）—— 来自 DeepMind Control Suite
- **GO1**（Unitree Go1 四足机器人）
- **Humanoid**（人形机器人）—— 来自 DeepMind Control Suite

它们都依赖本体感受（关节位置、速度等）进行控制，目标是在平坦或者复杂地形上稳定、快速地前行，并且尽量避免摔倒。

Walker 和 Humanoid 是来自 DeepMind Control Suite 的经典任务，包括了 stand、walk、run 三个难度递进的任务。

本章节的目的是让大家能够详细深入了解这三类机器人的强化训练过程，了解三种不同形态的 agent 同样完成相同的 RL 训练行走过程中的相同点和不同点，并且由于形态、奖励设计、控制策略等的差异，训练难度和方法都会有些不同。

## 11.1 机器人行走任务的共性与差异

### 11.1.1 共性

三个任务无论是行走、稳定站立还是奔跑，这个训练过程都有一个统一确定的 RL 框架，并且是连续控制的过程，而不是离散的。

**统一的 RL 训练框架**概括为：先看周围（观测维度中的状态）→ 进行决策动作 → 这个动作在物理引擎算一步（仿真）→ 系统告诉你是否做得好（奖励）或是否失败了（终止）。

在 MotrixLab 里，每一帧环境都会返回一个观测向量，这就是策略"看周围"的唯一信息来源。

对于行走任务，观测通常不包含视觉，而是**本体感受式**的，比如：
- 所有关节的当前位置
- 关节速度
- 身体基座的线速度和角速度
- 重力方向在基座坐标系下的投影
- 上一帧的动作，帮助感知自己刚才做了什么
- 命令速度

这些不同来源的信息会被拼接成一个一维浮点数数组，作为策略网络的输入。

策略网络拿到观测向量后，会输出一个动作分布。因为我们做的是连续控制，所以策略通常输出的是一个多维高斯分布的均值与标准差，然后在这个分布里采样得到具体的动作向量。

动作 action 被应用到机器人的关节上之后，MotrixSim 物理引擎就会执行一个仿真步，解算整个机器人的刚体动力学，算出下一时刻的身体位置、关节角度、接触力、碰撞情况等等。

三个机器人的物理模型不一样，但 MotrixLab 对它们的仿真接口完全一致：都是"给定动作 → 推进仿真 → 返回新状态"。

仿真结束后，环境会根据当前状态计算出一个即时奖励，并判断是否触发终止条件。

整个"观测→动作→仿真→奖励"的循环，在不同机器人之间，只是维度数字和奖励权重不同，循环本身的代码结构与数据流动方向是完全统一的。正因为有这个统一，MotrixLab 才能用一个 `train.py` 脚本，只通过切换环境名 `--env walker`、`--env go1`、`--env humanoid`，就完成三个任务的训练。

### 11.1.2 差异

差异主要来自三件事：**机器人形态、环境复杂度、奖励与终止设计**。

**1. 机器人形态差异（2D 双足 vs 3D 人形 vs 4足）**

| 维度 | Walker | Humanoid | GO1 |
|:---:|:------:|:--------:|:---:|
| 自由度 | 6 | 21 | 12 |
| 动作维度 | 6 | 21 | 12 |
| 观测维度 | 24 | 59 | 48 |
| 形态 | 2D 平面 | 3D 空间 | 3D 空间四足 |

- Walker 自由度少、观测少，动作维度固定，为最简单的行走任务
- Humanoid 自由度非常高（21维），动作多，目标姿态区域为强约束的非凸空间
- GO1 动作维度居中，但需处理四足步态协调（如 trot、pace、bound）

**2. 环境从简到繁递进**

| | Walker | Humanoid | GO1 |
|:--:|:------:|:--------:|:---:|
| 地形 | 标准 DM 平地 | 标准 DM 平地 | 多种地形（平/崎岖/随机） |
| 任务目标 | 速度跟踪 | 速度跟踪 | 速度指令跟踪 |

**3. 奖励结构差异**

- **Walker**：以站稳为主，再乘一个前进奖励，结构简单易学
- **Humanoid**：posture × speed × energy × gait 的**乘法结构**，任何一项差都会明显拉低总回报，学出来更"像人走路"，但训练更难
- **GO1**：大量加权项（速度跟踪、姿态、扭矩、动作变化率、四足腾空时间等）

**4. 终止条件差异（严格程度不同）**

- **Walker**：主要以数值异常（NAN）终止，比较宽松
- **Humanoid**：头高、躯干直立度、速度极值、数值有效性多条件终止，更严格
- **GO1**：机身躯干与地面接触即终止，更贴近四足防摔逻辑

**5. 任务驱动方式差异（速度参数 vs 速度指令）**

- **Walker / Humanoid**：通过固定 move_speed 定义任务难度档位
- **GO1**：每次 reset 随机采样速度指令 commands，策略学的是"步态跟踪"

## 11.2 稳定性、接触、平衡与协调控制

要理清训练过程完整的数据流向：任务 cfg 被注册 → 通过装饰器 registry 被调用 → 该类被存进全局字典中。同理任务环境类、RL 算法也是这样被注册的。随后我们启动训练，装饰器被调用，系统会自动判断推理框架。

RL cfg 的修改会影响 RL 的训练过程（训练算法本身的超参数），改任务的 cfg 会影响环境本身（比如训练的目标、奖励等）。

## 11.3 复杂运动任务中的训练挑战

不同任务的训练时间根据复杂程度的不同而不同。

**1. 奖励权重设计不合理**
以 2D Walker 举例，如果修改站立和前进的奖励比重，可能会**学不会前进的行为，只会在原地打转**。

**2. 观测空间维度不够**
减少关键的观测空间的信息，可能会出现**训练很慢以及训练后的效果不稳定**的情况。

**3. 终止条件过松或过严**
- 过松：agent 学不会策略，一直摔倒
- 过严：agent 不敢尝试，一直在原地，几乎学不了任何东西

**4. 神经网络大小不匹配**
- 太小的神经网络容易陷入局部最优，对于长程任务来说容易找不到最优解
- 太大的神经网络容易过拟合，泛化能力变差
