# 注册系统

## 注册系统的作用

MotrixLab 有两套注册表，分工不同：

### 1. `motrix_envs.registry`（仿真环境）
- 用字符串 `name`（例如 `"cartpole"`）绑定：环境配置类（`EnvCfg` 子类）和环境实现类（`ABEnv` 子类）。
- 对外统一入口是 `make(name, sim_backend=..., env_cfg_override=..., num_envs=...)`
- 作用：把「环境名字」解析成可运行的环境对象，并保证「先注册配置、再注册实现」的顺序。

### 2. `motrix_rl.registry`（RL 训练配置）
- 仍用同一个环境名字 `env_name`，在框架名（当前固定为 `"skrl"`）和训练后端（`"jax"` / `"torch"`）下注册 PPO 等训练超参的 dataclass。
- `default_rl_cfg(env_name, "skrl", backend=...)` 会按后端优先选专用配置，否则回退。
- 作用：把「环境名字 + 训练后端」解析成默认 RL 超参。

## 环境配置、环境实现、RL 配置的注册方式

示例：如何让以 `cartpole` 为最小闭环（名字在三处一致）：

推荐目录结构（与仓库一致）：

```
motrix_envs/src/motrix_envs/basic/cartpole/
├── cfg.py              # ① 环境配置 + @registry.envcfg("cartpole")
├── cartpole_np.py      # ② 环境实现 + @registry.env("cartpole", "np")
├── cartpole.xml        # MotrixSim 模型
└── __init__.py         # from . import cartpole_np  # 触发注册

motrix_rl/src/motrix_rl/cfgs.py
└── class basic:
        @rlcfg("cartpole")   # ③ RL 配置，名字必须与上面一致
        class CartPolePPO(PPOCfg): ...
```

三处字符串必须相同：`"cartpole"`。

`train.py` 里 `--env cartpole` 会依次走：
1. `motrix_rl.registry.default_rl_cfg("cartpole", "skrl", backend=...)`
2. `motrix_envs.registry.make("cartpole", sim_backend=..., num_envs=...)`

### 环境配置（`EnvCfg` + `@registry.envcfg("名字")`）

**写什么：** `cfg.py` 里定义继承 `EnvCfg` 的 dataclass，并挂上装饰器：

```python
@registry.envcfg("cartpole")
@dataclass
class CartPoleEnvCfg(EnvCfg):
    model_file: str = model_file
    reset_noise_scale: float = 0.01
    max_episode_seconds: float = 10
    render_spacing: float = 2.0
```

**`validate()` 何时跑：** 注册时不会调用；只有在 `registry.make()` 里实例化配置后才会调用。

**装饰器做了什么：** `@registry.envcfg("cartpole")` 在类定义被执行时调用 `register_env_config`，在全局表 `_envs` 里写入：
`_envs["cartpole"] = EnvMeta(env_cfg_cls=CartPoleEnvCfg, env_cls_dict={})`（此时还没有实现类）。

### 环境实现（`ABEnv` / `NpEnv` + `@registry.env("名字", "np")`）

**写什么：** `cartpole_np.py` 继承 `NpEnv`，绑定同一名字和仿真后端：

```python
@registry.env("cartpole", "np")
class CartPoleEnv(NpEnv):
```

**装饰器顺序与依赖：** 必须先有 `@envcfg`，再有 `@env`。

**必须实现的三个抽象方法：**
- `apply_action(actions, state)`：将动作写进仿真
- `update_state(state)`：读新状态，计算 obs/reward/terminated
- `reset(data)`：重置环境

**如何让装饰器执行——import 链：**
```
import motrix_envs
  → basic/__init__.py: from . import cartpole, ...
    → cartpole/__init__.py: from . import cartpole_np
      → 执行 @envcfg（cfg 在 cartpole_np 里 from .cfg import）
      → 执行 @env
```

新任务：在对应分类的 `__init__.py` 里加 `from . import your_env_np`，否则注册了也找不到。

### RL 配置（`motrix_rl.cfgs` 里 `@rlcfg("名字")`）

```python
class basic:
    @rlcfg("cartpole")
    @dataclass
    class CartPolePPO(PPOCfg):
        max_env_steps: int = 10_000_000
        check_point_interval: int = 500
        policy_hidden_layer_sizes: tuple[int, ...] = (32, 32)
        value_hidden_layer_sizes: tuple[int, ...] = (32, 32)
        rollouts: int = 32
        learning_epochs: int = 5
        mini_batches: int = 4
```

**`@rlcfg` 的行为：**
```python
def rlcfg(env_name: str, backend: str = None):
    def decorator(cls):
        backends = ["jax", "torch"] if backend is None else [backend]
        for b in backends:
            _register_rlcfg(env_name, "skrl", b, cls)
        return cls
```

对 cartpole：未写 `backend` → 同一份 `CartPolePPO` 同时注册到 `skrl/jax` 和 `skrl/torch`。

## 总结：接入新任务的具体步骤

假设新任务注册名为 `my-task`：

### 步骤 1：实现并装饰
- `@registry.envcfg("my-task")`
- `@registry.env("my-task", "np")`

### 步骤 2：接上 import 链
A. 挂到现有分类（推荐）：在 `motrix_envs/basic/__init__.py` 增加 `from . import my_task`
B. 新建顶层分类：在 `motrix_envs/__init__.py` 增加 import

### 步骤 3：RL 配置
在 `motrix_rl/cfgs.py` 里：
```python
@rlcfg("my-task")
@dataclass
class MyTaskPPO(PPOCfg):
    ...
```

### 步骤 4：用同一字符串跑命令
```bash
uv run scripts/train.py --env my-task
uv run scripts/view.py --env my-task
```
