# 运动机器人 Demo 讲解：Walker / GO1 / Humanoid

## 核心目标

在 MotrixLab 的 `motrix_envs` 中，提供了多种预置的运动机器人环境，最典型的就是：
- **Walker**（双足步行者）
- **GO1**（Unitree Go1 四足机器人）
- **Humanoid**（人形机器人）

它们都依赖本体感受（关节位置、速度等）进行控制，目标是在平坦或者复杂地形上稳定、快速地前行，并且尽量避免摔倒。

---

## 一、机器人行走任务的共性与差异

### 1.1 共性

三个任务有一个确定统一的 RL 框架，是一个**连续控制**的过程，而不是离散的。

统一的 RL 训练框架一句话概括：先看周围（观测）→ 做出决策（动作）→ 物理引擎仿真 → 系统反馈（奖励/终止）。

在 MotrixLab 里，每一帧环境都会返回一个观测向量，这就是策略"看周围"的唯一信息来源。对于行走任务，观测通常不包含视觉，而是**本体感受式**的：
- 所有关节的当前位置
- 关节速度
- 身体基座的线速度和角速度
- 重力方向在基座坐标系下的投影
- 上一帧的动作
- 命令速度

**最关键的一点**：整个"观测→动作→仿真→奖励"的循环，在不同机器人之间，只是维度数字和奖励权重不同，循环本身的代码结构与数据流动方向是完全统一的。正因为有这个统一，MotrixLab 才能用一个 `train.py` 脚本，只通过切换环境名 `--env walker`、`--env go1`、`--env humanoid`，就完成三个任务的训练。

### 1.2 差异

差异主要来自三件事：**机器人形态、环境复杂度、奖励与终止设计**。

**1. 机器人形态差异（2D 双足 vs 3D 人形 vs 4足）**

| 机器人 | 维度 | 观测空间 | 动作空间 | 特点 |
|--------|------|----------|----------|------|
| Walker | 2D 平面 | 较小 | 较小 | 最简单，只需保持平面平衡 |
| GO1 | 3D 空间 | 中等 | 中等 | 四足稳定，可承载更多传感器 |
| Humanoid | 3D 空间 | 较大 | 较大 | 最复杂，双足平衡难度最高 |

**2. 奖励结构差异**

- **Walker**：以站稳为主，再乘一个前进奖励，结构简单易学
- **Humanoid**：`posture × speed × energy × gait` 的乘法结构，任何一项差都会明显拉低总回报，训练更难但学出来更"像人走路"
- **GO1**：大量加权项（速度跟踪、姿态、扭矩、动作变化率、四足腾空时间等）

**3. 终止条件差异（严格程度不同）**

- **Walker**：主要以数值异常（NAN）终止，比较宽松
- **Humanoid**：头高、躯干直立度、速度极值、数值有效性多条件终止，更严格
- **GO1**：机身躯干与地面接触即终止，更贴近四足防摔逻辑

**4. 任务驱动方式差异（速度参数 vs 速度指令）**

- **Walker/Humanoid**：通过固定 `move_speed` 定义任务难度档位
- **GO1**：每次 reset 随机采样速度指令 `commands`，策略学的是"步态跟踪"

## 二、稳定性、接触、平衡与协调控制

需要先理清训练过程完整的数据流向：

1. 任务 cfg 被注册——通过装饰器 `@registry.envcfg`，被存进全局字典
2. 同理任务环境类、RL 算法也是这样被注册的
3. 启动训练，装饰器被调用，系统自动判断推理框架
4. 训练开始后，环境与算法进行交互循环

## 三、复杂运动任务中的训练挑战

不同任务的训练时间根据复杂程度的不同而不同。

### 1. 奖励权重设计不合理

以 2D walker 为例：如果修改站立和前进的奖励比重，可能会学不会前进的行为，只会在原地打转。

### 2. 观测空间维度不够

减少关键的观测空间的信息，可能会导致训练很慢，以及训练后的效果不稳定。

### 3. 终止条件过松或过严

- 过松：agent 可能学不会策略，一直摔倒
- 过严：agent 不敢尝试，几乎学不了任何东西

### 4. 神经网络大小不匹配

- 太小的神经网络：容易陷入局部最优
- 太大的神经网络：容易过拟合，泛化能力变差
